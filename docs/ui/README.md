# UI 设计说明

本目录用于沉淀抗凝小助手的 UI 设计资料，与技术方案分开维护。

当前仓库通过 `/` 提供统一入口，通过 `/ui/` 提供可直接预览和部署的静态原型：

- 入口：`/ui/index.html`
- 逻辑：`/ui/app.js`
- 样式：`/ui/styles.css`
- 平台路由：
  - 规范路径：`/ui/wechat/home/`、`/ui/wechat/records/`、`/ui/wechat/inr/`、`/ui/wechat/me/`
  - 规范路径：`/ui/android/home/`、`/ui/android/records/`、`/ui/android/inr/`、`/ui/android/me/`
  - 规范路径：`/ui/ios/home/`、`/ui/ios/records/`、`/ui/ios/inr/`、`/ui/ios/me/`
  - 兼容路径：`/wechat/...`、`/android/...`、`/ios/...` 暂时保留，避免破坏旧线上链接。

## UI 当前范围

- 首页：今日服药、强提醒、最近 INR、下次检测、INR 趋势。
- 记录：每日服药时间线，按系统操作时间记录服药动作。
- INR：校准后/校准前双曲线、异常分层、检测记录。
- 我的/设置：检测方式、偏移量、检测周期、服药规则、通知、账号等。

## UI 验收矩阵

| 场景 | 必须展示/交互 | 验收标准 |
| --- | --- | --- |
| 首页状态卡片 | 今日服药、最近一次 INR 校正值、校准前原始值、下次检测时间、超明显提醒 | 首屏无需滚动即可看到最近 INR 与下次检测；提醒使用高对比卡片，不弱化到普通文案。 |
| 完成服药 | 点击“完成服药”后出现明日剂量确认 | 用户必须在“按计划服用”和“手动输入”中选择；手动输入展示片数输入框；不出现补服分支。 |
| INR 异常分层 | 目标范围外 ±0.1 内弱提示，超过 ±0.1 强提示 | 弱提示用次级状态色与说明文案；强提示使用醒目色、加粗标题和“联系医生/复测”边界提示。 |
| INR 展示 | 校正后值主显示，校准前值弱显示 | 首页、INR 列表、趋势图 tooltip/详情均以 correctedValue 为主，rawValue 仅作为小字号/次级色参考。 |
| INR 趋势 | 原始值与校正值两条曲线 | 图例同时展示“校准后/校准前”，两条曲线可区分，并显示目标范围参考。 |
| 设置 | 检测方式、偏移量、检测周期 | 检测方式可选；偏移量说明“只影响新记录”；周期支持 day/week/month + interval。 |
| 不做范围 | 多吃、补服 | 页面不提供“多吃规划”和“补服”功能；服药只记录用户操作时的系统时间。 |
| 文案来源 | 业务说明、状态、提示、按钮、标签尽量来自服务端 payload | 客户端只保留模块标题、固定导航和固定类型 key；首页提醒、INR 状态、检测方式说明、剂量确认文案从 `displayText` / `copy` 读取。 |
| 多语言准备 | 服务端返回 `locale` 与结构化 copy | 静态原型使用模拟服务端 payload，正式端按 API 字段渲染，避免在 Flutter/小程序中散落业务中文。 |

## 文案服务端化边界

本轮将静态原型中的业务文案集中到模拟服务端 payload：

- 可保留在客户端：模块标题、底部导航、平台名称、固定路由、固定枚举 key。
- 应由服务端返回：首页强提醒、最近 INR 文案、下次检测说明、完成服药后的明日剂量确认、INR 异常分层、检测方式/偏移量说明、帮助说明。
- 每个可变业务文案块应优先读取 `displayText` 或 `copy`；后续接入真实 API 时仅替换数据源，不改 UI 结构。

## 后续建议

正式开发时建议把当前静态原型拆分为：

- Flutter 页面组件规范。
- 微信小程序页面组件规范。
- 设计 Token：颜色、间距、字号、状态色、圆角、阴影。
- 交互规则：完成服药后确认明日剂量、INR 异常提醒、检测周期设置等。
